Unit Testing

Posted on Dec 27, 2025 DRAFT

Unit testing should focus on validating units of behavior, not units of code glued together by abstract interfaces and mocking frameworks.

For the final user, it does not matter whether the system has 95% coverage if it still fails in production. Coverage is a means to an end, not the goal itself. What matters is that observable behavior is correct under realistic inputs.

Testing implementation details instead of behavior leads to brittle tests that pass while the system is broken.

Functional Core Pattern

The most effective way to write unit tests without mocks is to separate the system into:

  1. Functional core: pure decision-making logic
  2. Mutable shell: I/O, side effects, integration code

Pure functions have:

  • no hidden inputs
  • no side effects
  • deterministic output

This makes them easy to test using simple input/output assertions.

// ❌ Naive design (hard to test)
pub fn handle_card_scan(card_id: &str) {
    let user = db_lookup(card_id);
    if user.is_authorized {
        unlock_door();
        log_event("access granted");
    } else {
        trigger_alarm();
        log_event("access denied");
    }
}
// ✅ Functional core + mutable shell
pub enum AccessDecision {
    Grant,
    Deny,
}

pub fn decide_access(
    authorized: bool,
    within_time_window: bool,
) -> AccessDecision {
    match (authorized, within_time_window) {
        (true, true) => AccessDecision::Grant,
        _ => AccessDecision::Deny,
    }
}
// Simple unit tests
#[test]
fn grants_access_when_authorized_and_on_time() {
    assert_eq!(
        decide_access(true, true),
        AccessDecision::Grant
    );
}

#[test]
fn denies_access_when_not_authorized() {
    assert_eq!(
        decide_access(false, true),
        AccessDecision::Deny
    );
}
// Mutable shell (thin and boring)
pub fn handle_card_scan(card_id: &str) {
    let authorized = db_lookup(card_id).is_authorized;
    let on_time = check_schedule(card_id);

    match decide_access(authorized, on_time) {
        AccessDecision::Grant => {
            unlock_door();
            log_event("access granted");
        }
        AccessDecision::Deny => {
            trigger_alarm();
            log_event("access denied");
        }
    }
}

Why not just use interfaces and then mock?

Interfaces (traits) with a single implementation are often a smell when introduced solely for testability.

They:

  • add indirection
  • hide data flow
  • require mocking frameworks
  • increase cognitive load

Most interfaces exist only to make testing possible which is a design smell.

Functional Injection via Closures

When you need to test logic that sometimes performs an effect (email, log, write, publish) you can still do it without defining a interface.

Instead of defining a full trait and mocking it, inject behavior, not structure.

In Rust, you do this via:

  • closures
  • function pointers
  • generics over functions

For example:

// ❌ Interface + mock approach
pub trait Notifier {
    fn notify(&self, msg: &str);
}

pub struct EmailNotifier;

impl Notifier for EmailNotifier {
    fn notify(&self, msg: &str) {
        send_email(msg);
    }
}

pub fn process_event(n: &dyn Notifier, critical: bool) {
    if critical {
        n.notify("alert!");
    }
}
struct MockNotifier {
    called: RefCell<bool>,
}

impl Notifier for MockNotifier {
    fn notify(&self, _: &str) {
        self.called.set(true);
    }
}

#[test]
fn test_alert_is_sent() {
    let mock = MockNotifier { called: RefCell::new(false) };
    process_event(&mock, true);
    assert!(mock.called.get());
}
// ✅ Functional injection via closures
pub fn process_event<F>(critical: bool, notify: F)
where
    F: Fn(&str),
{
    if critical {
        notify("alert!");
    }
}

#[test]
fn test_alert_is_sent() {
    let mut captured = None;

    process_event(true, |msg| {
        captured = Some(msg.to_string());
    });

    assert_eq!(captured.unwrap(), "alert!");
}

This does not replace traits everywhere. This is best when behavior is small and local.

Traits are still valid when behavior is shared across modules, multiple real implementations exist and the boundary is I/O, hardware, protocol, etc. For async or long-lived boundaries, traits remain the correct abstraction.

Key takeaway: If an interface exists only so you can mock it, it should probably be a closure.

Testing with Real Managed Dependencies

You should not mock managed dependencies, things you control, such as database, message broker, SQL queries.

Mocking these gives false confidence. Mocks do not fail on schema drift, serialization bugs, etc.

Use real instances in integration tests. Run local temporary database container. Spin up Postgres via Podman in CI and run migrations before tests.

Yes, tests and setup will be slower. But these catch real regressions.

Data-Driven Polymorphism with Enums

Yes, even though this is a unit testing post its fundamental to understand this point to write better code.

Prefer enums + match over traits for polymorphism inside your system.

This keeps the world closed, explicit and traceable.

Example:

pub enum Message {
    Email { body: String },
    Sms { text: String },
}

pub fn route(msg: Message) -> &'static str {
    match msg {
        Message::Email { .. } => "Routing to SMTP",
        Message::Sms { .. } => "Routing to Gateway",
    }
}

#[test]
fn test_routing() {
    let msg = Message::Sms { text: "Hi".into() };
    assert_eq!(route(msg), "Routing to Gateway");
}

This avoids pointer chasing, exhaustive testing, and explicit behavior.

Enums are ideal when you own all variants. If downstream users need to extend behavior then traits are the correct tool.

Analogy

Testing with interfaces and mocks is like hiring actors to perform a play about a factory to see if the factory works.

Testing with a functional core and concrete data is like taking a real component off the assembly line, putting it on a test bench, and measuring exactly how it reacts to real electricity.

What to do when I have interfaces/traits?

Traits are still valid when:

  • the world is genuinely open
  • implementations are multiple and independent
  • the boundary is external (I/O, hardware, network)

Examples:

  • MQTT client
  • database driver
  • filesystem abstraction
  • hardware interface

To test them correctly:

  1. unit-test the functional core behind the trait
  2. integration-test the real implementation
  3. avoid mocking the trait itself

Example:

pub trait Publisher {
    fn publish(&self, topic: &str, payload: &[u8]) -> Result<(), Error>;
}

Bad test: mock Publisher and assert it was called. Good tests: test message construction as a pure function, test MqttPublisher against a real broker.

Traits should be thin, at the edge of the system and rarely mocked.

Good unit tests are a consequence of good design.

TL;DR

  • Pure logic => unit tests
  • Effects => integration tests
  • Behavior injection => closures
  • Open world => traits
  • Closed world => enums
  • If a trait exists only for mocks => redesign
